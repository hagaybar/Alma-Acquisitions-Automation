=============================================================================
KEY FINDINGS AND LESSONS LEARNED
=============================================================================

1. MANDATORY PROCESSING STEP:
   ⚠️ Invoices MUST be processed/approved before payment
   - Without processing: ERROR 402459 "Error while trying to retrieve invoice"
   - Correct sequence: Create → Process → Pay
   - API operation: POST /invoices/{id}?op=process_invoice

2. INVOICE STATE MACHINE:
   Created (ACTIVE/InReview/PENDING/NOT_PAID)
     ↓ process_invoice
   Approved (ACTIVE/Ready to be Paid/APPROVED/NOT_PAID)
     ↓ paid
   Paid (CLOSED/empty/APPROVED/PAID)

3. POL AUTO-CLOSURE CONDITIONS:
   - Requires BOTH: invoice paid AND item received
   - In this test: Invoice paid ✓, Item received ✗
   - Result: POL remains SENT (not closed)
   - Next step: Receive item to trigger POL closure

4. INVOICE ALERTS AFTER PROCESSING:
   - ADDITIONAL_INVOICE_EXISTS: Multiple invoice lines for same POL
   - DIFFERENT_VENDOR_ACCOUNT: Vendor account mismatch warning
   - NOT_READY_LINES: Some lines still in review
   (Alerts don't block payment but indicate potential issues)

5. FUND AUTO-EXTRACTION:
   - Successfully extracted AAC_FUND from POL fund_distribution
   - No manual fund code specification needed
   - Fund distribution: 100% AAC_FUND (25.0 ILS)

6. INVOICE LIFECYCLE:
   - Unprocessed invoices may be deleted by system after retention period
   - Best practice: Process invoices soon after creation
   - First invoice (35925649570004146) was no longer accessible
